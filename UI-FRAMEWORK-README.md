# LearnPlaywright ŌĆö Enterprise Automation Framework

A production-grade, data-driven E2E testing framework built with **Playwright** and **TypeScript**. Designed to scale to thousands of test cases across multiple browsers, environments, and user roles ŌĆö with zero test-file pollution.

---

## What makes this different

Most Playwright setups dump everything into the test file: browser logic, environment resolution, skip conditions, credential management. This framework pulls all of that out.

A test file in this project contains *only* business logic. Everything else ŌĆö which browser to use, which environment to target, which credentials to load, whether to run at all ŌĆö is declared in an Excel registry and resolved automatically before the first test runs.

## How execution control works

### The Registry (`testRegistry.xlsx`)

Every test case has a row in the Registry sheet. The columns determine exactly how it runs:

| Column | What it controls |
|---|---|
| `TestCaseID` | Unique ID ŌĆö maps this row to a test file |
| `Description` | Human-readable title shown in the report |
| `Priority` | P0 / P1 / P2 / P3 ŌĆö used in filtering and reporting |
| `TestType` | system-testing / integration / e2e / regression |
| `Run` | `TRUE` / `FALSE` ŌĆö master on/off switch |
| `Environment` | `QA` / `PROD` ŌĆö which environment to run against |
| `Browser` | `chromium` / `firefox` / `webkit` / `all` |
| `Tags` | `@smoke`, `@critical`, `@regression` ŌĆö for filtering |
| `UserRole` | `standard` / `admin` / `locked` ŌĆö which credential to use |

Disabling a test is a one-cell change. No `test.skip()` scattered across files.

### Pre-execution project filtering

Standard Playwright setups spin up every configured browser and skip irrelevant tests at runtime. If three browsers are configured but a test only runs on Firefox, you get `2 skipped, 1 passed` ŌĆö which clutters your report and wastes runner time.

This framework reads the Registry *before* Playwright initialises its workers. `getActiveProjects()` in `playwright.config.ts` builds the `projects` array dynamically ŌĆö only the browsers actually referenced by enabled tests get a worker. A test that specifies `browser: firefox` simply never appears in the Chromium or WebKit worker at all.

The report shows `1 passed`. Clean.

## Credential resolution chain

Credentials never appear in test files or the registry. The resolution chain is:

```
testRegistry.xlsx  ŌåÆ  UserRole: "standard"
       Ōåō
environments.json  ŌåÆ  envPassKey: "QA_STANDARD_PASS"
       Ōåō
.env               ŌåÆ  QA_STANDARD_PASS=the_actual_password
       Ōåō
ConfigManager      ŌåÆ  returns the resolved password to the test
```

Adding a new environment is three steps: add a row to `environments.json`, add the key to `.env`, update the registry row.

npx playwright test --shard=3/10

# Run tests matching a tag (uses search ŌĆö pass as grep)

## Environment variables

Copy `.env.example` to `.env` and fill in values. The keys must match the `envPassKey` entries in `data/environments.json`.

| Variable | Required | Description |
|---|---|---|
| `QA_STANDARD_PASS` | Yes | Password for the standard QA user |
| `QA_LOCKED_PASS` | Yes | Password for the locked QA user |
| `QA_PROBLEM_PASS` | Yes | Password for the problem-behaviour user |
| `PROD_ADMIN_PASS` | Yes | Admin credentials for production |
| `ENV` | No | Overrides the environment for every test (default: from registry) |
| `WORKERS` | No | Parallel worker count (default: 4, CI-aware) |
| `TEST_TIMEOUT` | No | Per-test timeout in ms (default: 30000) |

## Adding a new test

1. **Add a row** to `testRegistry.xlsx` ŌĆö fill in all required columns.
2. **Add a data row** to any relevant sheet (e.g. `Login`, `ProductPage`) with the same `TestCaseID`.
3. **Create the test file** in the appropriate `tests/` subfolder:

```typescript
import { test, expect } from '../../base/baseTest';

test('CartCheck-TC05: Add item and verify total', async ({ page, tcData }) => {
    // tcData is already resolved ŌĆö browser, env, credentials, page data
    // Write only business logic here
    await page.goto('/cart');
    expect(await page.title()).toContain('Cart');
});
```

4. Set `Run = TRUE` in the registry to enable it.

That's it. No browser configuration, no `.skip()`, no credential management in the test file.

## Scaling to 5,000+ tests

The framework is built with large suites in mind:

**O(1) registry lookups.** `RegistryCache.ts` builds a `Map<tcId, TestCaseData>` once per worker on first access. Every subsequent lookup in `baseTest.ts` is a direct Map get, not a linear scan.

**Pre-indexed sheet maps.** `globalSetup.ts` indexes every Excel data sheet into a Map before processing. At 5,000 tests across 5 sheets, this eliminates ~25,000 linear searches.

**No compile-time inference.** The registry is loaded with `fs.readFileSync` at runtime, not as a static `import`. TypeScript never has to infer types from a 5,000-row JSON file.

**Horizontal CI scaling.** Use `--shard=N/M` to split the suite across parallel CI jobs:

```yaml
# .github/workflows/test.yml (example)
strategy:
  matrix:
    shard: [1, 2, 3, 4, 5]
steps:
  - run: npx playwright test --shard=${{ matrix.shard }}/5
```

**Configurable workers.** Set `WORKERS=8` (or higher) via environment variable to parallelise within a single machine without code changes.

## Custom HTML reporter

Every run generates a self-contained HTML report at `custom-reports/<ProjectName>-<timestamp>.html`. No server needed ŌĆö open it in any browser.

The report includes:

- **KPI cards** ŌĆö total, passed, failed, skipped, flaky at a glance
- **Browser breakdown chart** ŌĆö stacked pass / skip / fail per browser
- **Priority distribution** ŌĆö dynamic doughnut from actual run data
- **Tag coverage** ŌĆö horizontal bar showing which tags ran most
- **Filterable results table** ŌĆö search by ID, title, tag, status, priority; filter buttons with live counts
- **Trace links** ŌĆö direct links to Playwright trace files on failure
- **Flaky detection** ŌĆö flaky tests are badged and searchable

Configure the reporter in `playwright.config.ts`:

```typescript
reporter: [
    ['./utils/CustomReporter.ts'],
    ['html', { open: 'never' }],  // keep Playwright's built-in as a backup
],
```

## CI/CD ŌĆö GitHub Actions

```yaml
name: Playwright Tests

on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main]

jobs:
  test:

      - run: npm ci
      - run: npx playwright install --with-deps

      - name: Run tests (shard ${{ matrix.shard }}/4)
        run: npx playwright test --shard=${{ matrix.shard }}/4

          QA_STANDARD_PASS: ${{ secrets.QA_STANDARD_PASS }}
          QA_LOCKED_PASS:   ${{ secrets.QA_LOCKED_PASS }}
          QA_PROBLEM_PASS:  ${{ secrets.QA_PROBLEM_PASS }}
          WORKERS: 4

      - uses: actions/upload-artifact@v4
        if: always()
        with:
          name: report-shard-${{ matrix.shard }}
          path: custom-reports/
          retention-days: 30
```

Store all `*_PASS` variables as repository secrets under **Settings ŌåÆ Secrets and variables ŌåÆ Actions**. Never hardcode them.

## Project structure decisions

**Why Excel for the registry?**
QA teams live in Excel. Non-engineers can enable, disable, and reconfigure tests without touching TypeScript. The Excel file is the test plan and the execution config in one document.

**Why a nested JSON structure?**
Flat registries hit a wall when tests need different data for different pages. The nested structure (`data.Login`, `data.ProductPage`) mirrors the page object hierarchy and scales naturally as new pages are added.

**Why a custom reporter instead of the built-in one?**
The built-in Playwright reporter shows individual browser runs as separate rows. This framework groups them ŌĆö one row per logical test case, with per-browser status columns side by side. That's the format QA managers and stakeholders can actually read.

## Security notes

- `.env` is listed in `.gitignore` and must never be committed
- `environments.json` contains only credential key *names*, not values
- `globalSetup.ts` validates all `envPassKey` values are present in `.env` before the first test runs ŌĆö a missing key fails loudly at startup, not silently mid-suite
- All test data flowing into the HTML reporter is HTML-escaped before DOM insertion

## Troubleshooting

**`testRegistry.json not found`**
Run `npx playwright test` once to trigger `globalSetup.ts` which generates the JSON from Excel. If the Excel file is missing, place `testRegistry.xlsx` in `data/`.

**All tests showing as skipped**
Check the `Run` column in your Excel registry. All values must be `TRUE` (not `true`, `1`, or `Yes` ŌĆö Excel boolean `TRUE`).

**`Missing .env key: QA_STANDARD_PASS`**
Copy `.env.example` to `.env` and fill in all values. `globalSetup.ts` will list every missing key at startup.

**Wrong environment running**
`ConfigManager.ts` falls back to `QA` and logs a warning if `ENV` is unset or misspelled. Check the console output at the start of the run ŌĆö the warning includes the value it received and the available options.

**Browser mismatch ŌĆö test not running on expected browser**
The `Browser` column in the registry must exactly match one of: `chromium`, `firefox`, `webkit`, or `all`. Check for trailing spaces in the Excel cell.
